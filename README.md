## Reflection

### Adoption Of Secure Coding Standards

Throughout this course, I learned that security must be integrated from the beginning of development rather than treated as a final step. By adopting secure coding standards such as input validation, memory safety practices, and proper exception handling, I was able to reduce vulnerabilities early in the development lifecycle.

Applying standards such as SEI CERT C++ and OWASP Secure Coding Practices helped ensure consistency, reliability, and compliance across all coding activities.

---

### Risk Evaluation And Cost-Benefit Analysis

I developed an understanding of how to evaluate risks based on likelihood and severity. High-risk vulnerabilities such as SQL injection and memory corruption were prioritized for immediate remediation due to their potential impact on system security and data integrity.

I also learned that early remediation significantly reduces long-term costs, while delayed fixes increase the risk of breaches, system failure, and financial loss.

---

### Zero Trust Security

The Zero Trust model reinforced the principle that no user or system should be trusted by default. Every access request must be verified, authenticated, and authorized before being granted.

In my work, this concept aligns with the implementation of strict input validation, access control mechanisms, and continuous monitoring, ensuring that systems remain secure even in hostile environments.

---

### Implementation Of Security Policies

Through Project One and Project Two, I developed and presented a comprehensive security policy that integrates:

* Defense in Depth (DiD)
* Triple A Framework (Authentication, Authorization, Accounting)
* Encryption strategies (AES-256, TLS 1.2/1.3)
* DevSecOps automation tools (Cppcheck, SonarQube)

These policies provide a structured approach to identifying vulnerabilities, prioritizing risks, and implementing mitigation strategies across the SDLC.
